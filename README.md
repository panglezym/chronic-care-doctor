# 慢愈 · 医生端

**Chronic Care AI — Doctor Portal Prototype**

> 面向内分泌科、心内科、全科医学等慢性病主治医生的协同管理 Web Portal 原型，共 9 个页面，与患者端 App 形成完整的医患闭环。

本仓库包含一个完整的高保真静态原型，在浏览器中直接打开即可查看所有页面。

```
chronic-care-doctor.html   # 全部 9 页，单文件原型
```

## 产品定位

医生端解决的核心问题：

> 一位主治医生管理 10–50 名慢病随访患者，复诊间隔往往长达 1–3 个月。大量异常在两次复诊之间发生，医生无从得知；等到下次见面时，数据已经碎片化，无法还原完整病程。

慢愈医生端让医生能够：

```
实时感知  →  精准干预  →  闭环确认  →  AI 提效
```

- **实时感知**：Dashboard + 预警中心，高风险患者立刻可见
- **精准干预**：消息 + 医嘱，不用等复诊，随时远程处置
- **闭环确认**：患者在 App 确认医嘱，状态实时同步到医生端
- **AI 提效**：临床 AI 助手基于完整患者档案给出循证建议，节省查阅时间

**目标用户**：内分泌科 / 心内科 / 全科医学，主治医师至副主任医师，习惯使用 HIS 系统。

## 设计系统

### 设计哲学

> **深色侧边栏 = 角色感知。内容区克制 = 数据优先。**

医生端与患者端共享设计 Token，但在以下维度有意做出区分：

| 维度 | 患者端（温暖） | 医生端（专业） |
|------|--------------|--------------|
| 侧边栏 | 无侧边栏，Tab Bar | 深色 `#1A1714`，240px 固定 |
| 布局密度 | 宽松，大字号 | 紧凑，信息密度高 |
| 数据呈现 | 单指标突显 | 多维度并排（表格/多图） |
| 情绪基调 | 被关注，有成就感 | 高效，掌控全局 |

## 核心功能说明

### AI 临床助手

医生端 AI 与患者端 AI 调用相同的后端接口，但系统提示词完全不同：

| 维度 | 患者端 AI | 医生端 AI |
|------|----------|----------|
| 语言风格 | 通俗易懂，口语化 | 临床术语，循证导向 |
| 输出格式 | 建议+鼓励，短段落 | 方案A/B对比，含注意事项 |
| 知识库 | 患者健康常识 | ACC/AHA 2023 / ADA 2024 / ESC 2023 |
| 上下文 | 个人数据+用药+医嘱 | 同上 + eGFR / UACR / LDL-C 等检验值 |
| 边界 | 严重情况提示联系医生 | 标注"以上为AI辅助，最终由临床判断" |

右侧患者上下文面板实时显示：档案摘要 / 今日关键指标 / 历史会话记录，医生无需切换页面即可参考。

### 医嘱状态流转

```
医生创建
    ↓
[待确认]  → 推送 App 通知给患者
    ↓         患者在提醒页点击确认
[已确认]
    ↓
[进行中]  → 患者每日执行
    ↓
[已完成]  → 到期或医生手动关闭
```

患者确认后，医生端医嘱列表状态实时更新；若患者 24 小时未确认，医生端显示"待确认"提醒。

### 报告审阅闭环

```
患者端                          医生端
  │                               │
  │  每月 25 日 AI 自动生成报告    │
  │                               │
  │ ←── 报告推送 ─────────────────┤  待审阅 +1
  │                               │
  │                               │  医生查看报告
  │                               │  填写批注
  │                               │  点击"批注并发送"
  │                               │
  │ ←── 批注推送 ─────────────────┤
  │                               │
月度报告页显示医生批注            报告状态 → 已审阅
```

### 预警分级处置

| 级别 | 颜色 | 触发条件示例 | 操作按钮 |
|------|------|------------|---------|
| 高风险 | 🔴 红色左边框 | 收缩压 > 160，连续 3 天超标 | 发消息 + 查看 |
| 中风险 | 🟡 琥珀色左边框 | 指标超目标范围，依从率 < 70% | 查看 |
| 信息 | 🔵 蓝色左边框 | 月度报告待审阅，复诊提醒 | 审阅 |

## 设计决策记录

<details>
<summary>为什么侧边栏用深色而不是白色？</summary>

患者端和医生端共用同一个后端，但两端的**用户身份**和**心理状态**截然不同：

- 患者：需要被关怀，温暖的暖白色调降低焦虑
- 医生：需要快速判断，深色侧边栏强化"工作台"感知，与患者端的轻量感产生明确区分

当两端同时打开时（医生也可能有自己的慢性病），深色侧边栏让医生立刻知道"我现在用的是工作端"。这是一个角色感知设计，而非纯美学决策。

</details>

<details>
<summary>为什么 AI 临床助手放在导航而不是内嵌到患者详情页？</summary>

两种方案都考虑过：

**方案 A（内嵌）**：在患者详情页右侧放 AI 对话框。优点是上下文最近；缺点是页面过于拥挤，医生无法专注于 AI 对话。

**方案 B（独立页面，最终选择）**：AI 助手有独立页面，但顶部可切换当前患者上下文。优点是对话空间充足，可以展开完整的临床分析；缺点是多了一次导航跳转。

选择方案 B 的核心原因：医生使用 AI 的场景通常是**深度思考**（评估方案、生成摘要），不是快速查一个数字。深度思考需要沉浸的空间，而不是侧边的小窗口。

</details>

<details>
<summary>报告审阅为什么不直接让患者发给医生，而是 AI 先生成？</summary>

传统流程：患者手写记录 → 复诊时带来 → 医生翻阅。

慢愈流程：AI 自动聚合一个月数据 → 生成标准化报告 → 医生审阅 + 批注 → 发回患者。

这个设计的关键点不是"自动化"，而是**标准化**。每一份报告的结构完全一致（达标率/趋势/AI建议），医生不需要花时间整理信息，可以直接进入判断。批注功能让医生的专业意见以结构化方式传递给患者，比口头复诊更可追溯。

</details>

<details>
<summary>消息中心为什么采用两栏布局而不是全屏对话？</summary>

医生在使用消息功能时，往往同时管理多个患者的消息。全屏对话（如微信式）需要频繁返回列表，切换成本高。

两栏布局（会话列表 + 对话区）让医生在看完一个患者消息后，眼睛向左扫一下就能看到下一个未读消息，**不需要任何额外导航动作**。

这是刻意借鉴了 Slack 的信息架构，而非聊天软件的架构。医生的工作场景更接近"处理工单"，而不是"和朋友聊天"。

</details>
